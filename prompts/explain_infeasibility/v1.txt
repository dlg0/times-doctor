- The excerpts may already include the most relevant ranges. Cite from them.

What to produce (in order):
1) Executive summary (3–6 bullets)
- State the most likely root cause(s) in plain language.
- Include the minimal conflicting components (constraints/variables/sets) you believe create the contradiction.

2) Root cause analysis with evidence
- For each root cause, provide:
  - The specific constraints/rows or variables that conflict (names/patterns and affected indices: region, timeslice, vintage, year, commodity, process).
  - The mechanism of infeasibility (examples: mutually exclusive bounds on the same var; fixed activity combined with capacity/availability = 0; flow shares summing > 1 or < required; negative demand with binding lower bound; commodity balance with insufficient supply; ramping/seasonal profiles inconsistent with bounds; impossible emissions caps with fixed activity; unit/currency mismatches).
  - Evidence citations using the format:
    [QA_CHECK: L{{start}}-{{end}}]; [RUN_LOG: L{{start}}-{{end}}]; [LST: L{{start}}-{{end}}]
  - Include values when available (RHS, LB/UB, activity/fixings, duals/infeasibilities).

3) Minimal conflicting set (MCS) hypothesis
- List 3–7 most critical equations/variables that, together, are sufficient to make the model infeasible.
- For each, give a one-line reason it participates in the contradiction and cite lines.

4) Concrete remediation plan (step-by-step)
- For each root cause, give exact edits a modeller should make, referencing typical Veda tables and keys:

  - What to change (add/remove/relax/fix/align units; ensure non-negative bounds; remove contradictory FX with UP/LO; unify currencies/timeslices; fix negative demands or missing supplies; correct share group sums; enable required imports/supplies; adjust caps to feasible levels).
  - The precise keys to search for (Process, Commodity, Region, Year, Vintage, Timeslice, Group) derived from the cited equations/rows.
  - Provide example row(s) to add/modify/remove (use clear pseudo-rows if exact sheet rows are unknown).
- Order the fixes from lowest-risk/most-likely to resolve → highest-risk.

5) Quick validation steps
- Checklist to re-run and validate:
  - Expected messages disappearing from QA_CHECK and RUN_LOG.
  - Model status transitions to feasible; inspect objective; sanity-check flows/balances for the affected indices.
- If not enough info remains, instruct how to re-run with IIS/Conflict Refiner and what artifacts to capture:
  - CPLEX: rerun with conflict refiner/IIS enabled and include the conflicting rows/variables report in LST/log.
  - Gurobi: compute IIS and include the IIS report details.

6) Assumptions and unknowns
- Explicitly list any assumptions you made due to missing context or truncated excerpts.
- List the minimal extra information needed if uncertainty remains.

- Reference concrete equation/row/variable names and index patterns.
- Always cite evidence using [QA_CHECK: ...], [RUN_LOG: ...], [LST: ...] with line ranges.
